## 안전 경계

- 파괴적 DB 작업(DELETE, UPDATE, ALTER)은 실행 전 확인 필수
- 프로덕션 대량 데이터 변경은 테스트/스테이징에서 먼저 검증
- 테스트 실패 시 기본은 구현 결함을 의심한다. 단, 테스트의 가정·기댓값이 잘못되었다는 근거가 있으면 사유를 보고하고 테스트를 수정한다.
- 테스트 데이터, 픽스처, API 계약 변경 시 승인 필수
- 비밀정보(API 키, 토큰, `.env` 등)는 출력·로그·커밋에 절대 포함하지 않는다.
- 새 의존성 추가 전 승인 필수. 외부 네트워크/프로덕션 API 호출은 실행 전 확인한다.
- 사용자 명시 요청 없이 커밋·푸시·PR·이슈 코멘트 등 외부 가시 액션을 실행하지 않는다.

## 계획 관습

- 검증 없는 추측으로 행동하지 않는다. 불가피한 가정은 근거와 함께 명시한다.
- 트레이드오프를 명확히 밝힌다. 더 간단한 접근이 있으면 제안하고, 정당한 사유가 있으면 사용자의 요청에 반대 의견을 제시한다.
- 작업 방향·범위를 바꿀 수 있는 모호성은 즉시 질문한다. 사소한 표현 모호성은 가정을 명시한 뒤 진행한다.

## 작업 관습

- 문제를 해결하는 최소한의 코드만 작성한다. 추측에 기반한 코드는 배제한다.
- 요청된 범위만 수정하고, 주변 코드 정리는 별도 요청 시에만 수행한다.
- 요청 범위 외 추상화·유연성·예외처리·재작성을 추가하지 않는다. 동일 로직이 3회 미만 반복되면 추출하지 않는다.
- 커밋에서 구조적 변경(리네이밍, 추출)과 동작적 변경(기능 추가/수정)을 분리한다.
- 작업과 무관한 데드 코드를 발견하면 보고하되 직접 삭제하지 않는다.
- 본인의 수정으로 인해 불필요해진 임포트, 변수, 함수는 제거한다.

## 품질 가드레일

- 기존 코드베이스의 함수와 패턴을 검색하여 재사용 우선
- 규모에 맞는 에러 처리 적용(내부 코드와 프레임워크 보증은 신뢰)
- 안정적이고 널리 채택된 라이브러리 우선 활용 (허용: MIT/Apache-2.0/BSD/ISC. 회피: GPL/AGPL/LGPL 등 카피레프트)
- 성공 기준을 정의한다. 검증될 때까지 반복한다.
- 작업을 검증 가능한 목표로 변환한다
  - "유효성 검사 추가" → "잘못된 입력에 대한 테스트 작성 후 통과 확인"
  - "버그 수정" → "버그를 재현하는 테스트 작성 후 통과 확인"
  - "X 리팩토링" → "리팩토링 전후의 테스트 통과 확인"
- 다단계 작업의 경우 간략한 계획을 수립한다:
  1. [단계] → 검증: [확인 사항]
  2. [단계] → 검증: [확인 사항]
  3. [단계] → 검증: [확인 사항]
  - 성공 기준이 명확해야 독립적인 작업이 가능하다. "작동하게 만들기"와 같은 모호한 기준은 불필요한 재질의를 야기한다.
