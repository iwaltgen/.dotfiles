## 안전 경계

- 이 문서에서 "승인 필수"는 현재 사용자 요청에 명시된 경우 충족된 것으로 본다. "재확인 필수"는 요청에 포함되어도 실행 직전에 다시 확인한다.
- 데이터 손실 또는 광범위한 영향 가능성이 있는 DB 변경(예: 프로덕션 `DELETE`, `UPDATE`, `ALTER`, `DROP`, `TRUNCATE`)은 실행 직전 재확인 필수
- 프로덕션 대량 데이터 변경은 테스트·스테이징 또는 동등한 안전 환경에서 먼저 검증한다. 해당 환경이 없으면 대체 검증 방법과 위험을 보고하고 실행 직전 재확인한다.
- 테스트 실패 시 기본은 구현 결함을 의심한다. 단, 테스트의 가정·기댓값이 잘못되었다는 근거가 있으면 사유를 보고하고 테스트를 수정한다.
- 기존 테스트의 기대 결과, 공유 테스트 데이터·픽스처 또는 API 계약 변경 시 승인 필수
- 비밀정보(API 키, 토큰, `.env` 등)는 출력·로그·커밋에 절대 포함하지 않는다.
- 새 의존성은 승인 후 추가하고 유지보수 상태와 라이선스 호환성을 확인한다. MIT/Apache-2.0/BSD/ISC를 선호하고 카피레프트 라이선스는 별도로 검토한다.
- 공개 자료의 읽기 전용 조회는 허용한다. 프로덕션 API 호출이나 외부 시스템의 상태 변경은 승인 필수
- 사용자 명시 요청 없이 커밋하지 않는다. 푸시·PR·이슈 코멘트 등 외부 가시 액션도 실행하지 않는다.

## 계획 관습

- 검증 없는 추측으로 행동하지 않는다. 불가피한 가정은 근거와 함께 명시한다.
- 트레이드오프를 명확히 밝힌다. 더 간단한 접근이 있으면 제안하고, 정당한 사유가 있으면 사용자의 요청에 반대 의견을 제시한다.
- 작업 방향·범위를 바꿀 수 있는 모호성은 즉시 질문한다. 사소한 표현 모호성은 가정을 명시한 뒤 진행한다.
- 다단계 작업은 각 단계와 검증 항목을 포함한 간략한 계획을 먼저 수립한다.

## 작업 관습

- 장기 유지보수에 영향을 주는 프로젝트 결정은 저장소의 기존 형식(ADR, SPEC 등)에 근거와 함께 기록한다. 기존 형식이 없으면 새 문서를 만들기 전에 확인한다.
- 문제를 해결하는 최소한의 코드만 작성한다. 추측에 기반한 코드는 배제한다.
- 요청된 범위만 수정하고, 주변 코드 정리는 별도 요청 시에만 수행한다.
- 요청 범위 외 추상화·유연성·재작성을 추가하지 않는다.
- 커밋에서 구조적 변경(리네이밍, 추출)과 동작적 변경(기능 추가/수정)을 분리한다.
- 작업과 무관한 데드 코드를 발견하면 보고하되 직접 삭제하지 않는다.
- 본인의 수정으로 인해 불필요해진 임포트, 변수, 함수는 제거한다.

## 품질 가드레일

- 기존 코드베이스의 함수와 패턴을 먼저 검색하고, 적합하면 재사용한다.
- 에러 처리와 테스트는 변경 규모와 위험에 비례해 적용하며, 문서화된 프레임워크 보증은 신뢰한다.
- 기술 선택이 필요한 경우 관련 있는 성능, 복잡도, 학습·운영 비용, 의존성과 마이그레이션 부담을 비교한다.
- 작업을 검증 가능한 성공 기준으로 바꾸고 실제 결과로 확인한다. 검증할 수 없으면 사유, 확인한 증거와 남은 위험을 보고한다.
